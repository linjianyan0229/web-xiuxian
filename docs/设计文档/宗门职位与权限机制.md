# 宗门职位与权限机制

> **状态：一期已实装（2026-07-10）**——`sect_members` 表 + 14 职位枚举（`node/src/utils/sectPositions.js`）+ 任免/逐出/传位已上线：人事权 rank≤3、神子对长老级以上任免 403（罢免信未实装暂直接拒绝）、定额职位默认编制内置枚举（建宗自定编制待 `sect_quota`）、峰主/真传弟子随山峰系统二期开放。**二期待实装**：山峰、执法堂与先办后奏、三类文书、宗主审判与投票、大牢、设施分层（§4~§8）。
> **适用范围**：一个宗门内部的身份等级、各身份职责与可操作范围（管理权 + 设施使用权 + 执法/罢免流程）。
> **本版已据反馈定稿多数规则**，仅余少量边界待确认（见 §11）。弟子最高层统一称 **真传弟子**（"核心弟子"即此，不再另设一层），且**真传弟子按山峰归属**（各峰各有其真传弟子）。

## 1. 职位体系总览

宗门内共 **14 种职位**，由高到低分层：

```
最高层（掌全宗）    太上长老 ── 宗主
                        │
储君层（近全权）      宗门神子 / 神女
                        │
山峰层（掌一峰）      峰主
                        │
长老级              藏经阁阁主 · 宗门仓库管理长老 · 任务长老
（掌一务 / 一门 /   内门长老 · 外门长老 · 执法堂执事(执法堂)
  执法）
                        │
弟子层（由高到低）  真传弟子(各峰) → 内门弟子 → 外门弟子 → 杂役弟子
```

### 1.1 职级次序（用于「长老级以上」判定、执法范围、投票资格）

数字越小职级越高：

| 职级 | 职位 | 归类 |
| :---: | --- | --- |
| 1 | 太上长老 | 最高层（对宗主有监督权，见 §5） |
| 2 | 宗主 | 最高层 |
| 3 | 宗门神子 / 神女 | 储君层 |
| 4 | 峰主 | 山峰层 |
| 5 | 藏经阁阁主 / 宗门仓库管理长老 / 任务长老 / 内门长老 / 外门长老 / **执法堂执事** | **长老级**（平级） |
| 6 | 真传弟子（各峰） | 弟子层 |
| 7 | 内门弟子 | 弟子层 |
| 8 | 外门弟子 | 弟子层 |
| 9 | 杂役弟子 | 弟子层 |

> **「长老级以上」= 职级 ≤ 5**，即所有非弟子职位（含**执法堂执事**）。据反馈，**执法堂全员皆为「执事」，身份位列长老级以上**；其实际裁决权归属**宗主与神子/神女之间**（见 §4）。神子/神女的罢免限制（§3）以「长老级以上」为界。

### 1.2 一句话定位

| 职位 | 定位 | 管辖 / 职权 |
| --- | --- | --- |

## 2. 管理权限

### 2.1 全宗管理（太上长老、宗主、神子/神女）

掌管整个宗门运转的 **6 大模块**：

| 管理模块 | 含义（预期能力） | 可委任的专职职位 |
| --- | --- | --- |
| **宗门信息管理** | 改宗门名称/简介/头像/背景/入门境界要求等 | —（仅最高层/神子） |
| **宗门仓库管理** | 公共仓库存入/取出/发放/调配资源 | 宗门仓库管理长老 |
| **宗门人员管理** | 招收/逐出成员，**任免各级职位**，调整弟子身份 | —（仅最高层/神子；神子有罢免限制） |
| **宗门任务管理** | 创建/发布/审核/结算宗门任务 | 任务长老 |
| **宗门藏经阁管理** | 维护藏经阁各层内容、兑换目录与准入层级 | 藏经阁阁主 |
| **宗门大牢人员管理** | 关押/释放/处置大牢中的人员 | 执法堂（见 §4） |

- 三者对下级管理权**向下涵盖**（可越峰、越门、越务直接处置）。
- 太上长老较宗主**额外**拥有对宗主的监督与审判发起权（§5）；日常管理权二者对等。

### 2.2 专职长老（掌一务，各一人）

由太上长老/宗主任命，各只握 6 大模块中对应的**一项**：

- **藏经阁阁主** → 藏经阁管理。
- **宗门仓库管理长老** → 宗门仓库管理（含审批真传弟子的资源申请）。
- **任务长老** → 宗门任务管理。

### 2.3 山峰与门务管理，及山峰归属

- **峰主**：管理**本峰弟子名单**，任命本峰真传弟子；权限仅限本峰。
- **内门长老**：管理**内门弟子**（考核/升降/调配，限内门）。
- **外门长老**：管理**外门弟子与杂役弟子**。
- **山峰归属规则**（据反馈，管辖为**转移式、非双头**）：
  - **外门弟子 / 杂役弟子无山峰归属**（隶外门，由外门长老管）。
  - **弟子晋为内门弟子后**，先统归**内门长老**管理其名单；此后**峰主可从内门弟子中挑选纳入本峰**，使其成为**本峰弟子**或**亲传弟子（即真传弟子）**。
  - **一旦被峰主纳入本峰，该弟子即转归该峰峰主管辖，内门长老不再拥有其名单（列表）权限**——弟子从内门长老名单移出、进入峰主名单。
  - **真传弟子（亲传）** 由峰主自本峰弟子中擢选，属该峰。
  - **内门弟子只能归属一座山峰**；**被峰主驱逐出峰 / 主动退峰 = 退出宗门**（离宗，`sect_id` 清空），无"回归散修内门"一说。

✓=有权；✗=无权；「本峰/本门」=仅限管辖范围；「⚑罢免信」=无直接权、须走罢免信（§3）。

## 3. 宗门神子 / 神女（一人）

- **产生**：由**太上长老或宗主直接提拔**（不经投票/晋升阶梯）。全宗**仅一人**。
- **权限**：**等同于宗主/太上长老**（全宗 6 大模块 + 越级处置）。
- **唯一限制——不可直接罢免长老级以上**：对「长老级以上」人员（职级 ≤ 5，**含峰主、各长老、执法堂执事**）神子/神女**不能直接罢免**，只能**向宗主/太上长老提交「罢免信」**，由其定夺。
  - 对**弟子层**（职级 ≥ 6）可**直接**任免/处置（与宗主同）。
- **继任**：宗主被罢免后，神子/神女为**新宗主首选人选**（须经太上长老投票过半，见 §5）。

## 4. 执法堂与执法权限

执法堂成员**皆为执事**，身份属**长老级以上**；执掌门规与刑罚，**实际裁决权归宗主与神子/神女之间**。

### 4.1 查阅举报信
- **普通举报信**（针对非宗主对象）：由**执法堂全员联合查看**，系统**随机给各执事分配处理数量**。
- **针对宗主的举报信**：**任何人不可查看**，由系统内置统计并触发审判（§5）。

### 4.2 执法与处罚范围
- 执法堂**可对除宗主 / 太上长老以外的任何身份**（含神子、峰主、各长老、其他执事、弟子）**执法并设立处罚**。
- **宗主与太上长老不受执法堂处置**（宗主仅受 §5 罢免机制约束）。

### 4.3 先办后奏（核心约束）
- 执法堂拥有**先办后奏**之权：可**先执行处罚，再申报**。
- 执行后**必须向裁决者（宗主 / 神子·神女）申报**（提交「执法申报」）。
- **超时惩戒**：若**自执行起一日（24 小时）内未申报**，则对该执法人员：
  1. **撤销其所属权利**（剥夺执法堂执事身份/权限）；
  2. **撤销其在这一日内已执行的相关处罚事件**（相关处置回滚、恢复原状）；
  3. **将其打入大牢**，听候发落。

> 实装要点：每条执法处置落库为一条**执法事件**（执行人/对象/处罚/执行时刻/是否已申报/申报时刻）。到期未申报的判定与回滚参照打坐的「惰性结算」——相关接口处理前扫描"超 24h 未申报"事件，批量执行撤权 + 回滚 + 入狱。

### 4.4 执法堂自身的约束
- **执法堂执事若有违纪，由执法堂其他执事处置，或由宗主 / 太上长老处置**（执法堂不自处宗主/太上，但其成员本身可被同堂执事或最高层追责）。
- 神子/神女因执事属长老级以上，不能直接罢免，须走罢免信（§3）。

## 5. 宗主的监督、罢免与继任

宗主不受执法堂直接处置，其去留由**监督发起 / 举报累积触发 + 太上长老投票**决定。

### 5.1 启动条件（二者其一）
1. **太上长老监督发起**：太上长老拥有对宗主的监督权，可**主动发起宗主审判**。
2. **举报满额自动触发**：**针对宗主的举报信满 100 条**即自动启动审判。
   - 宗主举报信**任何人不可查看**，由系统内置统计；
   - **同一人在罢免机制执行前对宗主只可举报一次**（系统按举报人去重计数）。

### 5.2 表决人
- **由太上长老们投票**表决。
- **若宗门无太上长老**，改由**各峰主投票**。

### 5.3 表决结果（据反馈）

| 表决结果 | 处置 |
| --- | --- |
| **全票通过** | 宗主**就地裁决处死**，随即启动新宗主选举（§5.4） |
| **达三分之二（未全票）** | **剥夺宗主权限，打入大牢**，随即启动新宗主选举（§5.4） |
| **过半但不足三分之二** | 宗主**不去职，但打入大牢禁闭一段时间**；**禁闭时长由太上长老提出、再由太上长老投票决定**（无太上长老、由峰主表决时，亦由峰主提出并投票议定）；不触发新宗主选举 |
| **未过半数** | **取消本次罢免裁决**（宗主留任） |

> **禁闭档职权代理（据反馈）**：宗主禁闭期间，**所有职权交由神子/神女代劳**；代理期间**神子/神女仍不可直接罢免长老级以上**，须向太上长老申请（同 §3）。**宗主禁闭结束后即恢复全部职权**。禁闭档属申诫性惩戒，不剥夺宗主之位、不触发改选。

### 5.4 新宗主选举
- 由**太上长老投票**产生（无太上长老则峰主投票，同 §5.2 回退）。
- **首选神子/神女继任**：太上长老投票表决神子/神女是否继任；
- **若神子/神女未获超过半数同意，则从峰主中选取（底线）**。

## 6. 编制（人数）规则

| 类别 | 职位 | 名额规则 |
| --- | --- | --- |
| **创立时固定、不可更改** | 太上长老、峰主、内门长老、外门长老、执法堂执事 | 建宗时定死；「峰主数」即**山峰数**（一峰一峰主），此后不可增改 |
| **定额一人** | 宗主、神子/神女、藏经阁阁主、宗门仓库管理长老、任务长老 | 各恒 1 人 |

- 实装：建宗时把各固定/定额职位的**编制上限**写入宗门配置（如 `sect_quota`），**不提供修改接口**；任命前校验"在编数 < 上限"。

## 7. 设施使用权限（分层）

设施分层：藏经阁 `一/二/三层`、宗门商店 `一/二/三层`、闭关室 `一/二/三级`。**层级累进**（高层涵盖低层）。**权限分三级**（据反馈）：

| 权限级别 | 适用职位 | 藏经阁 | 宗门商店 | 闭关室 | 仓库资源申请 |
| --- | --- | :---: | :---: | :---: | :---: |
| **三级（顶）** | 太上长老、宗主、神子/神女、藏经阁阁主、仓库管理长老、任务长老 | ≤三层 | ≤三层 | 三级 | ✓ |
| **二级** | 峰主、内门长老、外门长老、执法堂执事 | ≤二层 | ≤二层 | 二级 | ✗ |
| 弟子·真传 | 真传弟子 | ≤三层 | ≤三层 | 三级 | ✓（申请调用） |
| 弟子·内门 | 内门弟子 | ≤二层 | ≤二层 | 二级 | ✗ |
| 弟子·外门 | 外门弟子 | 一层 | 一层 | 一级 | ✗ |
| 弟子·杂役 | 杂役弟子 | ✗ | ✗ | ✗ | ✗ |

- **任务接取**：全体弟子（真传/内门/外门/杂役）皆可。
- 反馈原文："长老级以上、神子以下（即峰主、内外门长老、执法堂执事），除专职长老外，均为二级权限"——故峰主/内外门长老/执法堂执事列**二级**；**专职长老（阁主/仓库/任务）单列为三级**（与最高层同）。此处峰主(职级4)设施级低于专职长老(职级5)属规则明列，非笔误。
- 弟子设施逐条对应：杂役=仅任务；外门=任务+藏经阁一层+商店一层+闭关一级；内门=任务+藏经阁≤二层+商店≤二层+闭关二级；真传=任务+藏经阁≤三层+商店≤三层+闭关三级+仓库申请。

## 8. 设施与文书概念（占位，内容后期单列）

以下系统本身未实装，此处只固定"分层/概念"接口，**具体内容后期单独出文档**：

- **藏经阁（三层）**：功法/心法/秘典兑换库（阁主维护）。
- **宗门商店（三层）**：以贡献/灵石兑换物资。
- **闭关室（三级）**：打坐/修炼加成场所（可挂钩「打坐机制」）。
- **宗门仓库**：公共资源池；三级权限者/仓库长老存取发放，真传弟子「申请调用」走审批。
- **宗门任务**：可接取任务源；任务长老创建审核。
- **宗门大牢**：关押违纪成员/俘虏，配合执法堂。
- **文书三类**：举报信（检举，宗主类满 100 触发审判且不可查看）、罢免信（神子→宗主/太上，罢免长老级以上）、执法申报（执法堂→宗主/神子，先办后奏事后报）。

## 9. 任命与晋升关系

**任免链**

- 太上长老 / 宗主：可任免除自身外一切职位；提拔神子/神女；受理罢免信与执法申报；太上长老另可发起宗主审判。
- 神子 / 神女：可直接任免/处置弟子层；罢免长老级以上须提**罢免信**。
- 峰主：本峰内任命真传弟子、把内门弟子纳入本峰、调配本峰名单。
- 内门长老 / 外门长老：分别管理内门 / 外门+杂役弟子的升降考核。
- 专职长老（阁主/仓库/任务）、执法堂执事：只管本务/执法，不参与任免。

**弟子晋升阶梯**：`杂役弟子 → 外门弟子 → 内门弟子 →（被峰主纳入某峰后）真传弟子`。

**唯一性约束（已定）**：一名成员在一宗内**只能担任一个职位，不允许一人多职**。

## 10. 权限模型与数据结构建议（供实装参考）

现状：成员仅靠 `users.sect_id` 归属宗门，无职位/山峰/执法/编制概念。建议如下（沿用项目"原生 SQL + 无 ORM"惯例）。

### 10.1 职位枚举（`node/src/utils/sectPositions.js`）

```
taishang_elder    太上长老        rank 1
sect_master       宗主            rank 2
divine_child      宗门神子/神女    rank 3
peak_master       峰主            rank 4
scripture_master  藏经阁阁主      rank 5
warehouse_elder   宗门仓库管理长老 rank 5
task_elder        任务长老        rank 5
inner_elder       内门长老        rank 5
outer_elder       外门长老        rank 5
enforcer          执法堂执事      rank 5
core_disciple     真传弟子        rank 6
inner_disciple    内门弟子        rank 7
outer_disciple    外门弟子        rank 8
menial_disciple   杂役弟子        rank 9
```
每职位附 `rank`（长老级以上判定/罢免/执法范围/投票资格）与 `facilityLevel`（3/2 及弟子分层）。

### 10.2 表结构建议

- **`sect_members`**：`id, sect_id, user_id(唯一), position, peak_id(可空，单值——一人只属一峰), joined_time`。内门弟子 `peak_id` 为空时归内门长老名单；被峰主纳入某峰后 `peak_id` 落定、转归峰主，**内门长老名单查询须排除 `peak_id` 非空者**（转移式管辖）。**驱逐出峰 / 退峰 = 退宗**：直接清 `users.sect_id` 并删除其 `sect_members` 行（走删成员流程，非仅清 `peak_id`）。
- **`sect_peaks`**：`id, sect_id, name, master_id(峰主), created_time`；峰数创立时定死。
- **`sect_quota`**（编制固定/定额，§6）：`sect_id, position, quota`；建宗写入，无修改接口。
- **`sect_reports`**（举报信，§4/§5）：`id, sect_id, reporter_id, target_id, is_sect_master_target(0/1), reason, status, created_time`；对宗主类**按 reporter 去重**，系统计数满 100 触发审判，且**不下发查看**；普通类随机分派各执事查看。
- **`sect_enforcements`**（执法事件，§4.3）：`id, sect_id, enforcer_id, target_id, penalty, executed_time, reported(0/1), reported_time`；超 24h 未申报惰性扫描 → 撤权 + 回滚 + 入狱。
- **`sect_dismissals`**（罢免信，§3）：`id, sect_id, proposer_id(神子), target_id, reason, status, created_time`。
- **`sect_master_trials`** / **`sect_votes`**（宗主审判与投票，§5）：记审判发起、选民（太上长老，空则峰主）、票选，按全票/≥⅔/未过半判处死/剥夺入狱/取消，及新宗主选举（神子首选，过半否则峰主）。
- 藏经阁/商店/闭关室/大牢/任务各自独立表，权限校验只读 `rank` + `facilityLevel` + 设施层级常量。

### 10.3 权限判定（能力常量 + 职位映射）

- 能力常量：`SECT_INFO_MANAGE / WAREHOUSE_MANAGE / PERSONNEL_MANAGE / TASK_MANAGE / SCRIPTURE_MANAGE / PRISON_MANAGE / ENFORCE / REVIEW_REPORTS / PEAK_ROSTER_MANAGE / INNER_DISCIPLE_MANAGE / OUTER_DISCIPLE_MANAGE …`，维护**职位 → 能力集合**映射（对应 §2.4）。
- 特殊约束：
```
罢免限制：divine_child 对 rank<=5 目标 → 只能建 sect_dismissals，不可直接改 position
执法范围：enforcer 处罚目标 rank>=3（除 太上长老/宗主）；裁决权归 宗主 + 神子
先办后奏：enforcement 超 24h 未 reported → 撤权+回滚+入狱
宗主审判：对宗主 sect_reports（按 reporter 去重）计数 >100 或 太上长老发起 → 开审
          选民=太上长老（空则峰主）；全票=处死、≥2/3=剥夺入狱、未过半=取消
新宗主：  神子首选（太上长老过半通过），否则峰主中选
编制：    sect_quota 无修改接口；一人一职（唯一约束）
设施级：  facilityLevel 3/2；弟子按 §7 分层
山峰籍：  仅 rank<=7(内门弟子)及真传可有 peak_id；外门/杂役 peak_id 恒空
```
- **核心数值与判定必须在后端**（前端只按能力隐藏入口，防篡改）。
