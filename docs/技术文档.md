# 文字修仙 · 技术文档

> Web 文字修仙小游戏，前后端分离架构。本文档描述系统架构、数据库设计、接口规范、核心游戏机制与开发规范。
> 最后更新：2026-07-08

---

## 1. 项目概述

一款以修仙为题材的网页文字游戏。玩家注册后以「道号」行走此界，通过修炼、打坐、签到积累修为，突破境界（含雷劫死亡风险），并可在世界频道与其他道友交流。系统内置丹药、境界、排行榜等数值体系，配套完整的后台管理端（用户/境界/丹药/系统配置管理）。

- **已实装玩法**：注册登录、每日签到、点击修炼、定时打坐（挂机）、境界突破（含雷劫身陨）、修行日志、今日修炼统计、排行榜、丹药背包（赠送/丢弃）、头像（上传/外链）、性别与角色立绘、世界频道聊天、宗门（列表/搜索筛选/创建/详情 + 后台管理）。
- **占位待实装**：丹药服用（药力/buff 结算）、功法/心法（修炼速度机制已备）、宗门加入与成员管理、洞府/炼丹等模块、灵石产出与战斗系统。

## 4. 快速开始

### 4.1 环境要求

- Node.js ≥ 20（后端热重载依赖 `node --watch`）
- MySQL ≥ 8.0（首次启动自动建库建表，无需手工执行 SQL）

### 4.2 配置

复制 `node/.env.example` 为 `node/.env` 并按需修改：

| 变量 | 默认 | 说明 |
|---|---|---|
| `NODE_ENV` | development | 设为 `production` 时启动前强制安全校验（见 §7.4） |
| `PORT` | 3000 | 后端端口 |
| `FRONTEND_ORIGIN` | 空 | CORS 白名单（逗号分隔多来源）；空=开发放开、生产从严 |
| `DB_HOST/DB_PORT/DB_USER/DB_PASSWORD/DB_NAME` | localhost/3306/root/空/web_xiuxian | MySQL 连接 |
| `JWT_SECRET` / `JWT_EXPIRES_IN` | 示例值 / 7d | 令牌签名密钥与有效期 |
| `ADMIN_USERNAME/ADMIN_PASSWORD/ADMIN_EMAIL` | admin/admin123456/admin@xiuxian.local | 默认管理员（仅首次播种） |
| `SMTP_HOST/SMTP_PORT/SMTP_SECURE/SMTP_USER/SMTP_PASS/MAIL_FROM` | 空/465/1/空/空/空 | 邮箱验证码发信；HOST 留空=未配置（开发验证码打到控制台，生产发码接口 503）；465 隐式 TLS 保持 SECURE=1，587/25 设 0 |

### 4.3 启动

```bash
# 后端（端口 3000）
cd node && npm install
npm run dev        # node --watch 热重载
npm start          # 普通启动

# 前端（端口 5173，/api 自动代理到 3000）
cd web && npm install
npm run dev        # 开发
npm run build      # 生产构建 → dist/
npm run preview    # 预览构建产物
```

健康检查：`GET /api/health` → `{"status":"ok","game":"web-xiuxian"}`。

> 排查端口占用：`netstat -ano | grep :3000`，必要时 `taskkill //PID <pid> //F`（npm 启动的子进程可能残留）。

## 5. 系统架构

```
浏览器 ── Vue3 SPA（Vite 构建）
   │  HTTP /api/*（开发经 Vite proxy；生产经反代同源）
   ▼
Express 5（node/src/app.js）
   ├── /api/auth    注册/登录/登出（登录统一入口，按 role 分流）
   ├── /api/user    游戏接口（authRequired）
   ├── /api/admin   后台接口（adminAuthRequired）
   └── /api/uploads 静态托管（头像等上传文件，长缓存）
   ▼
MySQL（mysql2 连接池，原生 SQL）
```

设计原则：

- **数值与判定全部在后端**：修炼收益、突破判定、雷劫掷骰、冷却校验等一律服务端结算，前端只做展示与交互，防客户端篡改。
- **无 WebSocket / 定时任务**：世界频道用轮询增量拉取；打坐到期用「惰性结算」（相关接口处理前顺带补结算），架构保持无状态、易横向扩展。
- **并发安全靠原子 UPDATE**：所有资源变更 SQL 的 `WHERE` 带状态守卫（冷却时间、境界、资源余量、打坐时段），`affectedRows=0` 即竞态失败，返回 409。

## 6. 数据库设计

### 6.1 初始化与迁移

`node/src/config/db.js` 的 `initDatabase()` 在服务启动时执行，顺序为：

1. `CREATE DATABASE IF NOT EXISTS`（用不带库名的连接）；
2. 创建连接池；
3. 逐表 `CREATE TABLE IF NOT EXISTS`（DDL 内联在 db.js，无独立迁移工具）；
4. `ensureColumn()` 幂等补列（查 information_schema 后 `ALTER TABLE ADD COLUMN`，用于老库升级）；
5. 种子数据：`seedRealms()`（境界 83 条）、`seedPills()`（丹药 475 种/1425 件）、`seedDefaultAdmin()`、`seedSystemConfigs()`——均幂等，表非空/键已存在则跳过；
6. `UPDATE users SET is_online = 0` 清残留在线态。

**新增字段**照 `ensureColumn` 模式加调用；**新增表**追加 DDL 与 `pool.query(CREATE_XXX)`。

### 6.2 表一览

| 表 | 用途 | 关键点 |
|---|---|---|
| `users` | 玩家与管理员（单表，`role` 区分） | 见 §6.3 |
| `realms` | 境界表（id 1=凡人 … 83=圣人） | 属性 + 晋级要求 + 雷劫死亡率 + 签到奖励区间；id 非自增，取自数据文件 |
| `pills` / `pill_grades` | 丹药与凡/灵/道三档品质 | `pills.id` 为字符串主键（如 `pill_fanren_cultivation`）；`effects` 为 JSON 列 |
| `user_pills` | 玩家丹药背包 | 唯一键 `(user_id, pill_id, grade)` 合并计数；赠送走事务转移 |
| `email_codes` | 邮箱验证码（注册/重置密码） | 一次一码：校验只认最新一条未用未过期，`verifyAndConsume` 原子核销；索引 `(email, purpose, id)` |
| `player_logs` | 修行日志（叙事文本） | 旁路写入（失败只告警）；索引 `(user_id, created_time)` |
| `user_daily_stats` | 每日修炼统计 | `(user_id, stat_date)` 主键；UPSERT 累计；供「今日修炼」面板 |
| `world_messages` | 世界频道消息 | id 即时间序 / 增量游标；索引 `(user_id, id)` 供发言冷却查询 |
| `sects` | 宗门（列表/创建/详情已实装） | 名称唯一键；头像/背景外链、简介、境界门槛（`realm_req` 大境界名 + `realm_req_rank` 数值门槛）、宗主、活跃度；成员关系记在 `users.sect_id`，人数实时聚合，无独立成员表 |
| `announcements` | 修仙公告 | 后台发布/编辑/删除；`title/content/pinned置顶/status(1发布0下架)`，索引 `(status,pinned,id)`；前台「通知」弹窗「修仙公告」页只取 status=1（置顶优先） |
| `friendships` | 好友关系/结交请求 | `status` 0=待通过/1=已结交，方向记 `requester/addressee`；唯一键 `(requester_id,addressee_id)`；是否好友查两个方向 |
| `user_blocks` | 拉黑（单向） | `user_id` 拉黑 `blocked_id`；唯一键 `(user_id,blocked_id)`；结交/私信前双向校验 |
| `private_messages` | 私信（点对点） | `receiver_read` 已读标志；索引 `(receiver_id,sender_id,id)`/`(sender_id,receiver_id,id)`；会话按双方 id 聚合，无会话表 |
| `system_configs` | 键值系统配置 | 后台可视化编辑；`value_type` 决定控件（bool 开关 / number 输入框） |

各表无外键约束，删除玩家时在 `userModel.deleteUser()` 中**手动级联**清理：`player_logs`、`user_pills`、`user_daily_stats`、`world_messages`、`friendships`、`user_blocks`、`private_messages`，并删除其本地头像文件；其**执掌的宗门先行解散**（`disbandSectsLedBy`：清全体成员 `sect_id` 后删宗门行）。

## 7. 鉴权与安全

### 7.1 登录与令牌

- **统一登录入口**：玩家与管理员都走 `POST /api/auth/login`（account 可为道号或邮箱）。认证成功签发 JWT，payload 含 `id/daoName/role`（role=1 签 `'admin'`，否则 `'user'`）。
- **认证与授权分离**：登录只认证身份；能否进后台由受保护接口校验 role 决定，没有独立的后台登录接口。
- 注册/登录会把新令牌写入 `users.access_token` 并更新 `login_time`、置 `is_online=1`。

### 7.2 单会话吊销

`authRequired` / `adminAuthRequired` 除 JWT 验签外，还比对请求令牌与库中 `access_token`：

- 登出（`clearLoginState`）清空 `access_token` → 旧令牌立即失效；
- 异地重新登录覆盖 `access_token` → 先前会话被挤下线；
- 不一致一律返回 401「登录状态已失效」。

### 7.3 中间件

- `middleware/auth.js authRequired`：`Authorization: Bearer <token>` → 验签 → 查用户（`status!==1` 返回 403）→ 比对 access_token → 挂 `req.user`（USER_SELECT 公开视图，不含 password/email_code）。
- `middleware/adminAuth.js adminAuthRequired`：额外校验令牌 `role==='admin'` 且库中该行 `role=1`；玩家令牌访问后台返回 403。

### 7.4 生产安全

`NODE_ENV=production` 时 `server.js` 启动前执行 `assertProductionConfig()`，任一不满足即打印原因退出：

- `JWT_SECRET` 非开发默认值且 ≥16 位；
- `DB_PASSWORD` 非空；
- `ADMIN_PASSWORD` 非默认口令（未配置则跳过默认管理员播种，避免弱口令后门）。

CORS：配置 `FRONTEND_ORIGIN` 则仅放行列出来源（带 credentials）；未配置时开发放开、生产 `origin:false` 从严。

### 7.5 上传安全（头像）

- multipart 仅收 `avatar` 字段，白名单 JPG/PNG/WebP/GIF、≤2MB；
- 扩展名按 mimetype 取，**不信任客户端文件名**；
- 落盘后读文件头做**魔数复核**（`sniffImageMime()`），与声明类型不符即删文件返 400，防伪造 MIME；
- 文件名 `u<用户ID>_<时间戳>.<ext>`，不复用旧名（可放心长缓存）；换头像/删号清理旧文件；
- 外链头像仅校验 http/https 与长度并存 URL，不做服务端代理拉取。

### 7.6 邮箱验证码与重置密码

- 验证码 6 位数字、10 分钟有效、**一次一码**：`email_codes` 新码插新行，校验只认「最新一条未用未过期」（`verifyAndConsume` 原子置 used，防重放/防并发复用）；注册码与重置码**用途隔离**（purpose 字段）。
- 发码限频：同邮箱同用途 60 秒一封（409）。注册用途要求邮箱未被占用，重置用途要求邮箱已注册。
- 发信走 `utils/mailer.js`（nodemailer）：模板 `buildCodeMail()`（水墨风 HTML + 纯文本降级）；`SMTP_HOST` 未配置时开发环境打印验证码到控制台、生产环境发码接口 503。
- 重置密码成功即 `resetPasswordById`：换哈希 + **清空 `access_token`**（所有现有登录随单会话吊销机制立即失效）+ 置离线。
- 已知薄弱点：未限制验证码错误尝试次数（靠 TTL+单次使用兜底）、发码无 IP 级限频——公网部署建议反代层限流。

### 7.7 其他约定

- 消息/输入类内容（世界频道等）服务端清洗控制字符并限长；前端一律用插值 `{{ }}` 渲染纯文本，天然防 XSS。
- SQL 全部参数化（`?` 占位）；动态列名走白名单映射（签到字段、突破消耗字段、后台可更新字段等）；分页 LIMIT/OFFSET 先 clamp 为安全整数再内联（mysql2 对 LIMIT 占位符的限制）。

## 8. API 参考

统一约定：请求/响应均 JSON（上传除外）；鉴权走 `Authorization: Bearer <token>`；错误统一 `{ error: "中文提示" }`，状态码 400（参数）/401（未登录/令牌失效）/403（无权限/禁用）/404（不存在）/409（状态冲突：冷却、并发、条件不足）。

### 8.1 认证 `/api/auth`

| 方法 | 路径 | 入参 | 返回 | 说明 |
|---|---|---|---|---|
| POST | `/register` | `{ daoName, email, password, gender?, emailCode }` | `{ token, user }` | 新用户恒为玩家；gender 1/2 缺省 1；悟性随机生成；emailCode 受配置 `register_email_code_enabled` 控制 |

## 9. 核心游戏机制

### 9.1 悟性（comprehension）

- 注册与后台建号时 `rollComprehension()`：99.5% 均匀取 1~10，0.5% 直接 15；后台编辑经 `clampComprehension()` 收敛 0~100。
- **修炼速度（机制已备，待功法系统接入）**：最终修炼速度% = 基础10% + 基础10%×悟性%（满悟性封顶 20%），只作用于功法/心法，不影响修为获取。计算函数 `utils/comprehension.js cultivationSpeedPercent()`。

### 9.2 修为修炼（点击修炼）

- 单次收益 = 当前境界圆满修为 `advance_exp` × 5%，四舍五入保底 1（`utils/cultivationGain.js`）；修为丹加成已留參数（未接入按 0）。
- 冷却 `cultivate_cooldown_seconds`；`applyCultivate` 原子 UPDATE 带冷却+打坐守卫防连点/入定分心。
- `cultivation ≥ advance_exp` 时圆满，接口 409、前台按钮切「突破」；`advance_exp=0` 的境界（仙王及以上）不可打坐修炼，靠道法晋级。

### 9.3 打坐（定时挂机修炼）

- 时长白名单：一炷香 30 / 半个时辰 60 / 一个时辰 120 / 两个时辰 240 分钟；**入定期间不可中断**——修炼/突破/再次打坐接口全部拦截，无取消接口，SQL 层亦有打坐守卫兜底。
- 期满一次性结算：每小时得圆满修为 × `meditation_gain_percent_per_hour`%，四舍五入保底 1，封顶到圆满。
- **惰性结算**：无定时任务；`settleDueMeditation()` 被 profile/修炼/打坐/突破/日志等接口在处理前调用，离线到期回来即补结算。

### 9.4 境界突破

按当前境界行 `requirement_type` 判定（境界行即描述晋级到 `next_realm` 的要求）：

| 类型 | 条件 | 成功后 |
|---|---|---|
| 经验型 | 修为 ≥ `advance_exp` | 扣除，溢出保留 |
| 道韵型 | 道韵 ≥ `dao_yun_required` | 扣除 |
| 道法型 | 道法 ≥ `dao_law_required` | **不消耗** |
| 终点（圣人） | 不可突破 | — |

含「雷劫」的类型先按 `tribulation_death_rate` 掷死亡：陨落则 `death_count+1`、对应资源折半、境界不跌。成功/陨落均原子 UPDATE（境界+资源守卫），写日志 `breakthrough`/`death`。免死药/突破丹待丹药服用系统接入。

### 9.5 每日签到

- 满 24 小时可签（DB 时钟判定），总开关 `sign_in_enabled`。
- 奖励按境界类型：经验/道韵型 → 对应资源 × 随机百分比（区间存于境界行 `sign_in_min/max_percent`，后台可调，0.1~3%）；道法型/终点 → 道法固定 +1。
- 签到修为不截断圆满（可能略超）；原子 UPDATE 防并发重复签到。

### 9.6 丹药体系

- 数据：475 种丹药 × 凡/灵/道三档 = 1425 件；效果为 JSON 数组（`target/type(percent|flat)/value/duration/hours/polarity`）。
- 背包 `user_pills`：同种同品质合并计数；赠送为事务转移（扣减+累加+回滚，校验目标为正常玩家且非自己），丢弃带数量守卫防超扣；双方写日志。
- **服用未实装**（药力/buff 系统未接入），玩家暂无正式获得途径（炼丹/奖励待做），`addUserPills` upsert 已预留。

### 9.7 世界频道

- 全服单频道，无 WebSocket：前端每 5 秒 `afterId` 增量轮询（id 即游标），页签隐藏跳过。
- 发言：≤200 字、控制字符清洗、**敏感词拦截**（见 §9.10，命中 400，先于冷却校验）、间隔 `world_chat_cooldown_seconds`（DB 时钟计）；不写修行日志。

### 9.8 宗门（列表/创建/详情）

- **创建**：固定耗费 5000 灵石（`SECT_CREATE_COST`，硬编码待挪配置）；名称 2~16 字全服唯一；一人一宗（已有宗门不可再立）。事务内「原子扣款（余额+无宗门守卫合一条 UPDATE）→ 建宗（重名 `ER_DUP_ENTRY` 译 409）→ 创建者入驻任宗主」，写日志 `sect_create`。
- **境界要求**：`realm_req` 存大境界名（空=无要求）供展示筛选；`realm_req_rank` 存该档最小 realms.id，为**加入校验预留**（加入玩法未实装）。⚠ `rank` 是 MySQL 8 保留字，SQL 别名一律用 `realm_rank`。
- **成员/人数**：记在 `users.sect_id`，人数实时 COUNT 聚合；解散（后台）与删宗主号（级联）均事务清空成员归属。
- **活跃度**：字段就绪、默认 0、列表按其倒序；产出机制未接入，暂由后台编辑调整。
- 头像/背景暂用外链 URL（同头像外链口径校验），空值前端回退首字宗徽/默认洞府图。设计文档：`资源文件/机制相关/宗门机制.md`，进度清单：`计划反馈/宗门系统进度.md`。

### 9.9 修行日志与每日统计

- `player_logs`：叙事文本日志（register/login/logout/sign_in/cultivate/meditate_start/meditate/breakthrough/death/pill_gift/pill_receive/pill_discard/avatar…）。**旁路功能**：`addLog()` 内部 try/catch 只告警不抛错，控制器在操作成功后调用；管理员登录/登出不写。**新增玩家操作记得补一条 addLog**。
- `user_daily_stats`：修炼次数/打坐秒数/获得修为按 `CURDATE()` 归日 UPSERT 累计（同为旁路）。挂点：修炼成功、打坐结算（时长整段计入出定当天）、签到（仅修为型奖励）。**新增修为来源记得补 bumpDailyStat**。

### 9.10 敏感词过滤

- **词库**：`node/src/data/sensitive-words.txt`（约 5 万条，随库部署）——合并自根目录 `Vocabulary/` 17 个源文件（本地素材源，gitignore）；合并规则：按「小写+全角转半角」归一化去重、剔除单字条目、跳过 `A=B`/`A,B` 组合规则行。**白名单** `sensitive-words-allow.txt` 剔除误伤游戏交流的独立词条（修炼/弟子/师父/功法/大法/测试/联系/网络），不影响完整词条命中（「大法」剔除后「法轮大法好」仍拦）。两份 txt 井号行为注释，直接编辑、重启生效。
- **匹配器** `utils/sensitiveWords.js`：字典树，`server.js` 启动 `initSensitiveWords()` 建树（约 76ms、堆 64MB，单条消息扫描 0.03ms）。归一化口径同合并规则（`ＦａＬunGong`≈`falungong`）；匹配途中跳过填充符识别「习*近*平」「法 轮 功」式变体（先查树边、无边才跳，词条自身含符号不受影响）。API：`scanSensitive(text)` → `{ hit, words, sanitized(命中区间打码*) }`、`containsSensitive(text)`。
- **接入点**：世界频道发言强制拦截（§9.7）；通用检测接口 `POST /api/user/sensitive/check` 供前端预检与后续玩法复用。**待接**：注册道号、宗门名称/简介等文本入口（直接调 `containsSensitive`）。

### 9.11 公告与通知

- **修仙公告**（`announcements` 表）：管理员在后台「公告管理」页发布/编辑/删除。字段 `title/content/pinned置顶/status(1发布0下架)`。玩家端 `GET /api/user/announcements` 只返回 status=1（置顶优先、再按 id 降序，各 30 条）。数据访问 `models/announcementModel.js`，前后台控制器同在 `controllers/announcementController.js`（`normalizeAnnouncement` 统一校验，标题 1~60 字、正文 ≤2000 字）。
- **通知请求**（系统通知）：复用 `player_logs`，只取玩家被动收到、需知悉的类型——当前含 `pill_receive`（丹药赠礼）、`friend_request`（收到结交请求）、`friend_accepted`（结交获应允），定义在 `announcementController.NOTICE_TYPES`；宗门邀请等实装后在此数组追加即可自动纳入。`playerLogModel.listLogsByTypes(userId, types, limit)` 按类型 IN 筛选。
- **前台入口**：首页顶栏 ✉ 按钮打开 `components/NotificationModal.vue`（水墨分页弹窗：修仙公告 / 通知请求，各带条数角标）。未读红点：前端以「最大公告id:最大通知id」签名与 localStorage `noticeSeen` 比对，进页背景 `loadNotices()` 驱动红点，打开弹窗即标记已读。数据由首页拉取传入，弹窗自身不发请求。

### 9.12 社交：好友 / 私信 / 拉黑

- **好友（结交）**：`friendships` 表一行记一对关系（`requester`→`addressee`），`status` 0=待通过/1=已结交；是否好友需查两个方向（`findBetween`）。发起结交写对方通知（`friend_request`）、应允写发起方通知（`friend_accepted`）；婉拒/断交删除关系行。控制器 `relationController.js`（含关系状态 `GET /relations/:userId`，供名片按钮态：结交/接受结交/待通过/已是道友）。
- **拉黑**：`user_blocks` 单向记录。**结交/私信前双向校验**（`eitherBlocked`，任一方拉黑对方即 403）；**拉黑即断交**（`deleteBetween` 清除既有好友与待通过请求）。
- **私信**：`private_messages` 点对点，`receiver_read` 记已读。会话由收发双方 id 聚合（`listConversations` 取每对象最新一条 + 我方未读数；`getConversation` 拉往来消息并将对方发来的标记已读）。发送复用敏感词拦截 + 控制字符清洗 + ≤200 字。控制器 `privateMessageController.js`。数据访问 `models/{friendshipModel,userBlockModel,privateMessageModel}.js`。
- **前台入口**：① 世界频道名片 `ChatUserModal` 右侧四按钮（结交/切磋[占位]/私信/拉黑，打开时拉 `apiRelation` 定按钮态）；② 独立路由页 `views/FriendsView.vue`（`/friends`，**IM 双栏**：左栏列表[好友/群聊占位/结交请求/黑名单四页签，好友行带未读角标]，右栏内嵌私聊区[消息滚动 + 顶部对方信息与断交/拉黑 + 底部发送，5 秒轮询、进会话标记已读]）；③ 私信会话弹窗 `components/PrivateChatModal.vue`（同款逻辑，仅世界频道名片「私信」按钮使用）；④ 首页「伙伴」导航红点角标 = 未读私信 + 待处理结交请求（进页拉取、30 秒背景刷新）。

### 10.1 路由与守卫（`src/router/index.js`）

| `/admin/dashboard|users|realms|pills|sects|announcements|configs` | 后台各页（AdminLayout 布局） | requiresAdmin（非管理员挡回 /home） |

**没有 `/admin/login`**：登录后 `LoginView` 按 `auth.user.role` 分流。守卫读 localStorage 的单一 `token` + `user.role`。

**路由页过渡动画**（`App.vue`）：`RouterView v-slot` + `Transition mode="out-in"`，`router.afterEach` 按方向选名——游戏内页面（`GAME_PAGES = ['home', 'sect', 'friends']`）前进左滑/返回右滑，其余路由轻淡入，首次进入/刷新无动画；`scrollBehavior` 切页回顶。**防宽高跳动三件套**：① `style.css` 的 `html { overflow-x: clip }`（整页 translateX 的视觉溢出不再触发横向滚动条）；② 游戏内页面与 HUD 同规**锁一屏 + 列表内滚**（body 永无纵向滚动条，页面间视口宽度一致，窄屏媒体查询放开整页滚动）；③ 页面背景**勿用 `background-attachment: fixed`**（transform 会使其失效致背景闪跳）。新增游戏内页面：路由名加入 `GAME_PAGES` 并遵守三件套。

### 10.2 状态与 API 层

## 11. 开发规范与惯例

1. **新增后端模块**：`routes/` 建子路由 → `routes/index.js` 挂载 → 控制器放 `controllers/`、数据访问放 `models/`；游戏数值判定必须在后端。
2. **原子 UPDATE 模式**：资源变更 SQL 的 WHERE 带全部前置条件（冷却/境界/余量/打坐守卫），以 `affectedRows` 判定竞态，失败返回 409 并回读最新状态。
3. **旁路功能模式**：日志（addLog）、每日统计（bumpDailyStat）等非主干写入内部消化异常，只 `console.warn`，绝不打断业务。
4. **白名单原则**：动态列名、枚举入参（性别、品质、签到字段、打坐时长等）一律白名单校验。
5. **分页**：mysql2 对 LIMIT 占位符有限制——先 clamp 为安全整数再内联，其余条件用 `?`。
6. **时间判定用 DB 时钟**（NOW()/CURDATE()/TIMESTAMPDIFF），避免应用服务器时钟漂移。
7. **前端报错**：新页面统一 `useToast().error(...)`；加载失败不阻塞主界面（try/catch 静默 + 占位文案）。
8. 改动机制/表结构/接口后**同步更新 CLAUDE.md**（AI 协作上下文）与本技术文档。

## 12. 部署要点

1. 生产 `.env`：`NODE_ENV=production` + 强随机 `JWT_SECRET` + 非默认 `ADMIN_PASSWORD` + `DB_PASSWORD`，否则启动即退出（见 §7.4）。
2. 前端 `npm run build` 产物 `web/dist/` 由任意静态服务器/反代托管；将 `/api` 反代到后端端口（同源部署无需 CORS；跨域部署配 `FRONTEND_ORIGIN`）。
3. `node/uploads/` 为运行时数据目录（头像等），部署时需持久化并纳入备份；迁移服务器时随库一起搬。
4. 数据库零手工初始化：首次启动自动建库建表播种；升级部署直接重启即可（ensureColumn/INSERT IGNORE 幂等）。
5. 服务重启会将全员 `is_online` 清零（在线态由登录/登出维护），属预期行为。
